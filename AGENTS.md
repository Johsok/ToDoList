# AGENTS.md

本專案 AI 協作指引。Agent 請依此文件執行任務。

## 語言與格式

| 類型 | 語言 |
|------|------|
| 與使用者溝通、說明、錯誤訊息 | 繁體中文 |
| 程式碼、技術識別字、檔名 | English |
| `commit message`、terminal 指令 | English |

## 工作方式（Vibe Coding）

- 只檢查與任務相關的檔案，不做無關的全專案掃描。
- 做小範圍、精準的修改；能用 5 行解決就不要改 50 行。
- 沿用專案既有的結構、命名與程式風格。
- 避免過度設計：不提前抽象、不為假想需求加層。
- 回覆與 diff 中不要貼出大量未變更的程式碼。
- **未經使用者同意，不得新增套件或依賴。**
- 可行時執行與變更相關的測試、`lint` 或 type check。
- **除非使用者明確要求，不得 `commit`、`push`，也不得修改 secrets、API keys、password、`.env`。**

## 程式風格

- 註解使用 **JSDoc**（`/** ... */`）。
- 優先編輯既有檔案，不隨意新增檔案。
- 不撰寫使用者未要求的文件（如額外 `README`）。

## 專案脈絡

| 項目 | 說明 |
|------|------|
| 主檔 | `index.html`（單檔 HTML + CSS + Vanilla JS） |
| 規格 | `SPEC.md` |
| 資料庫腳本 | `supabase/schema.sql` |
| 後端 | Supabase（CDN `@supabase/supabase-js@2`） |
| 部署 | GitHub Pages |

實作前若有疑問，以 `SPEC.md` 為準。

## 完成任務後的回覆格式

每次任務結束，回覆**僅包含**以下四項（使用繁體中文）：

1. **修改了哪些檔案** — 列出路徑
2. **修改了什麼** — 簡短說明變更內容與原因
3. **如何測試** — 具體步驟或指令
4. **風險或後續建議** — 若有；無則寫「無」

## 禁止事項

- 不要 unsolicited `commit` / `push`
- 不要修改或寫入 secrets、API keys、`.env`
- 不要刪除檔案（除非使用者明確同意）
- 不要引入新依賴（除非使用者明確同意）
- 不要做大範圍重構（除非使用者明確要求）

## Git 相關

僅在使用者**明確要求**時才執行 `git commit` 或 `git push`。  
`commit message` 使用 English，簡潔描述「why」。
